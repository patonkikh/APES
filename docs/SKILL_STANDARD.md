# APES Skill Standard — Engineering Playbook

**Версия:** 1.0  
**Дата:** 2026-07-01  
**Источник:** [task.md](../task.md) §8, §9, §11

---

## 1. Назначение

Каждый Skill — законченная методология одной инженерной задачи. Skill **не играет роль** — Skill **реализует процесс**.

---

## 2. Формат файла

- Один файл: `skill.md` (lowercase)
- Markdown, без YAML frontmatter
- Без платформозависимого синтаксиса
- Целевой объём: 150–300 строк (детали — в README.md skill)

---

## 3. Обязательные секции

Каждый skill.md **обязан** содержать все 8 секций в указанном порядке:

### 3.1 Purpose

Что выполняет Skill. Одна задача, один измеримый результат.

### 3.2 Workflow

Последовательность шагов. Нумерованный список с чёткими входами/выходами на каждом шаге.

### 3.3 Decision Rules

Правила принятия решений: если условие X → действие Y. Таблицы или нумерованные правила.

### 3.4 Validation

Чеклист проверки качества результата. Агент обязан пройти его перед завершением.

### 3.5 Anti-patterns

Типичные ошибки, которых нужно избегать. Конкретные примеры плохих результатов.

### 3.6 Best Practices

Проверенные практики из инженерной методологии (не общие советы).

### 3.7 Output Structure

Шаблон результата — markdown-структура или таблица, которую агент заполняет.

### 3.8 Next Skills

Какие Skills использовать далее в цепочке разработки. Имена должны совпадать с каталогами в репозитории.

---

## 4. Запрещено

- Role Play ("You are a senior product manager...")
- "You are..." в любой форме
- Вымышленные роли и персонажи
- Предположение неизвестных данных без явного запроса у пользователя
- Платформозависимый синтаксис (YAML frontmatter, Cursor-specific directives)
- Несколько несвязанных задач в одном Skill

---

## 5. Обязательное поведение Skill

Каждый Skill обязан направлять агента:

- выявлять противоречия во входных данных;
- выявлять неоднозначности и запрашивать уточнение;
- выявлять пропущенные зависимости;
- выявлять отсутствующие входные данные;
- проверять полноту результата по Validation;
- предлагать улучшения, если результат неполный;
- учитывать безопасность, масштабируемость, сопровождаемость (где применимо).

---

## 6. Чеклист ревью перед публикацией

```
Skill Review:
- [ ] Ровно одна задача в Purpose
- [ ] Все 8 секций присутствуют
- [ ] Нет Role Play / "You are..."
- [ ] Workflow — последовательные шаги с входами/выходами
- [ ] Decision Rules — конкретные if/then правила
- [ ] Validation — проверяемый чеклист (≥5 пунктов)
- [ ] Anti-patterns — ≥3 конкретных ошибки
- [ ] Output Structure — заполняемый шаблон
- [ ] Next Skills — согласованы с соседними Skills в цепочке
- [ ] 150–300 строк в skill.md
- [ ] README.md skill содержит пример использования
```
